Critical Priority

P0 — Critical Gaps

12 gaps scored 13–15. These block a smooth client transition and carry the highest revenue and churn risk. 2 newly identified marketplace gaps (WEB-006, WEB-007) added — both scored P0.

15
CHF-001 / GO-002 / CAD-004 / OPD-001 · Vendor + GO Users + Client Admin + Ops
Live Delivery Tracking (OnFleet) — 4 Systems
Zero delivery visibility across all four interfaces once an order leaves the kitchen. HUNGRY has zero OnFleet integration — this must be built entirely from scratch. Tim Lee is leading discovery on reliably sending driver location updates from the web app. This gap spans every user type simultaneously:
  • Vendor Dashboard (CHF-001): Vendors need live driver status to time prep and reduce pickup call volume. Currently zero visibility once an order is dispatched.
  • GO Users / Diners (GO-002): Basic order status — placed → sent to restaurant → delivered. Currently zero post-order feedback for diners. Table-stakes UX expected by all modern platforms.
  • Client Admin Dashboard (CAD-004): Live countdown timer + delivery status stages. Reduces "where is food" support queries and gives admins self-service visibility.
  • Internal / Ops (OPD-001): End-to-end delivery management — live driver location, status progression (active → accepted → pickup → dispatch → delivered), ETA countdown. The only Ops Dashboard gap — scored maximum.
ETA availability is a proven competitive differentiator used to win business.
Strategic
Highest-scored gap — all 4 interfaces affected
15
CHF-002 / ADM-001 · Chef Dashboard + Admin (Internal)
Expected Bag Count Visibility
Vendors delay or reject orders because they can't gauge volume — and AMs currently lack the setup tools to solve it. Two connected gaps, one solution:
  • Vendor Dashboard (CHF-002): Bag count = one checkout transaction per vendor, calculated on a rolling average — last 5 consecutive days for production; last 5 same-day-of-week for office. V1 build confirmed: HUNGRY can use entree + side counts as a proxy initially. Ops call confirmation (March 16): Faisal: "Vendors would react more poorly to losing bag counts than to losing ETA" — it's their primary tool for staff scheduling. 75–80% accuracy confirmed sufficient.
  • Admin Setup (ADM-001): AMs set expected headcount during client setup; system self-updates via rolling average. HUNGRY currently tracks counts internally but does not display or formalize them. Without this, the vendor-facing bag count cannot be populated — ADM-001 is the data source for CHF-002. Absence of this data is currently managed through muscle memory, causing over-scheduling and vendor churn.
Hard dependency for unlocking the Auto-Assigner (ADM-002).

CHF-003 / GO-003 · Chef Dashboard + Group Order
Dietary Notes — Vendor Dashboard + Order Labels
Two connected surfaces, one safety gap — user dietary flags need to reach both the vendor and the physical order label:
  • Vendor Dashboard (CHF-003): Permanent user-level dietary flags should be visually called out on the vendor dashboard at prep time. Strategic Decision Required

Marketplace — Is It Worth Building Now?

Leadership guidance needed before sprint planning. The new analysis reveals 3 P0 gaps, 4 P2 gaps, and 1 P3 gap for full marketplace parity — a significant engineering lift on top of existing Uncatering priorities.

7
Total HUNGRY marketplace orders in all of 2025 — from 1 client, ~$5K total, across 13 markets. Newer markets launched in the past two years never had marketplace activated due to low demand and strategic decisions.
$813K
HungerHub marketplace revenue (Apr 2025–Feb 2026) — potential additional revenue for HUNGRY post-transition
3 P0s
Critical gaps if we commit to marketplace — all STRATEGIC (long arc)
The Case For
  • 65 recurring adhoc clients generated $719K ($520K recurring) — a real, growing segment
  • WEB-001 (P0): Adhoc catering ordering already scoped and scored critical since day one
  • WEB-006 & WEB-007 now also P0 — menu mods + special instructions are must-haves if we build marketplace
  • HH marketplace is fully operational; HUNGRY entering this space could capture that volume cleanly
The Case Against
  • Only 7 HUNGRY marketplace orders in all of 2025 — from 1 client, ~$5K total, across 13 markets. Newer markets launched in the past two years never had marketplace activated due to low demand and deliberate strategic decisions.
  • All 3 marketplace P0s are STRATEGIC — no quick tactical wins, all require long delivery arcs
  • Marketplace builds will compete directly for engineering bandwidth with Uncatering P0s ($12.5M revenue)
  • Full marketplace parity = 8 additional gaps (3 P0 + 4 P2 + 1 P3) on top of everything else
💡 Recommendation

If marketplace can be punted as a fast follow, we recommend treating WEB-001, WEB-006, and WEB-007 as P1 gaps rather than P0 — to be sequenced in the sprint immediately after the core Uncatering P0s are closed. Adding these three to the P0 sprint would limit engineering bandwidth at a critical transition moment, and the majority of migrated HH clients are Uncatering clients who won't be affected by marketplace gaps in the first months of transition. A fast-follow marketplace sprint in Q3 would let us deliver a complete marketplace experience rather than a partial one, and give us time to validate whether HH's $813K in marketplace revenue actually migrates with the clients.

Leadership Decision Needed

Should HUNGRY prioritize Marketplace now, phase it for later, or deprioritize entirely given current bandwidth? This affects sprint sequencing for Q2–Q3.

Option A: Build Now
Treat WEB-001, WEB-006, WEB-007 as P0 alongside Uncatering. Extended timeline on other priorities.
Option B: Phase It
Commit to WEB-001 only in Q2. Hold WEB-006 & WEB-007 for Q3 or later as minimum viable marketplace.
Option C: Deprioritize
Move all WEB gaps to P2/P3. Revisit marketplace in 6 months with better volume data.
